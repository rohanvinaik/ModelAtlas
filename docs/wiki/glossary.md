# Glossary

**Canonical definitions for ModelAtlas concepts. Every term is defined once here; other pages link back rather than re-explain.**

---

## anchor

A semantic label from a shared vocabulary of ~170 terms. Each anchor belongs to one [bank](#bank) and describes a characteristic of a model: `instruction-following`, `GGUF-available`, `Llama-family`, `code-generation`.

Models link to the anchors that describe them. Two models sharing many anchors are semantically related without an explicit edge — similarity is [emergent](Emergent-Similarity), not stored.

Anchors track provenance: `bootstrap` (seed dictionary), `deterministic` (Tier 1 extraction), `pattern` (Tier 2 regex), `vibe` (Tier 3 LLM), `expansion` (Phase D dictionary growth).

## audit score

A per-model quality metric (0.0–1.0) computed during [Phase D](Extraction-Pipeline) by comparing LLM-assigned anchors against deterministic pattern matchers. Models scoring below 0.7 become healing candidates.

## bank

One of eight orthogonal semantic dimensions. Each bank has a [zero state](#zero-state) and supports signed traversal: negative toward simpler/general/earlier, positive toward specialized/derived/novel.

The eight banks: **ARCHITECTURE**, **CAPABILITY**, **EFFICIENCY**, **COMPATIBILITY**, **LINEAGE**, **DOMAIN**, **QUALITY**, **TRAINING**.

See [Signed Hierarchies](Signed-Hierarchies) for why these are signed rather than categorical.

## bank alignment

A scoring component in the [query engine](Query-Engine). For each bank where the caller specifies a direction, the model's [signed position](#signed-position) is compared. Aligned models score 1.0; opposed models decay toward 0. The scores are **multiplicative** — any zero kills the total.

## IDF weighting

Inverse Document Frequency applied to [anchors](#anchor). Rare anchors (e.g., `proof-assistant` on 12 models) count far more than ubiquitous ones (`decoder-only` on 17K models). Computed as `log(N / count_models_with_anchor)`.

Used in [anchor relevance](Query-Engine) scoring and [seed similarity](Query-Engine).

## navigate_models

The primary [query engine](Query-Engine) tool. The calling LLM decomposes a natural language query into structured parameters (bank directions, required/preferred/avoided anchors, seed model). ModelAtlas does deterministic scoring. Four indexed SQL queries instead of 18K per-model lookups.

## overflow metadata

A flat `(model_id, key, value)` table for data that doesn't decompose into the network: SHA hashes, creation dates, raw parameter counts, license strings, benchmark scores. In a perfect design this table would be empty.

The one exception is [vibe summary](#vibe-summary) — a single prose sentence per model.

## signed position

A model's location within a [bank](#bank), stored as `(sign, depth)`. For example, `(-1, 2)` means two steps in the negative direction from [zero](#zero-state). The signed position is `sign × depth`.

Signed positions give gradient scoring — a 3B model is *close* to 7B, not a binary mismatch. See [Signed Hierarchies](Signed-Hierarchies).

## spreading activation

Priority-queue Bellman-Ford propagation from seed models through the network. Traverses both explicit [links](#bank) and shared [anchors](#anchor). Activation decays by 0.8 per hop, maximum 3 hops. Bank scoping prevents semantic bleeding.

Used by the legacy search engine. The structured [query engine](Query-Engine) uses IDF-weighted Jaccard instead.

## vibe summary

A single prose sentence per model capturing its irreducible "feel." The only natural language stored in the network. Generated by LLM (either Smol-Hub-tldr from model cards, or qwen2.5:3b from structured prompts), selected by the summary selection pass, and validated by the quality gate.

## zero state

The default position in each [bank](#bank), placed at the **query-frequency mode** — the thing most people are looking for. Most queries resolve near the origin with minimal computation.

| Bank | Zero State |
|------|-----------|
| ARCHITECTURE | transformer decoder-only |
| CAPABILITY | general language model |
| EFFICIENCY | ~7B parameters |
| COMPATIBILITY | PyTorch + transformers |
| LINEAGE | base/foundational model |
| DOMAIN | general knowledge |
| QUALITY | established mainstream |
| TRAINING | standard supervised (SFT) |

See [Signed Hierarchies](Signed-Hierarchies) for why zero placement matters.
